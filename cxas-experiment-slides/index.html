Mission Log 1 — The Discovery

Figuring out the MCP from scratch

No documentation on tool schemas. No examples. Just a tools/list endpoint and 60+ tools to enumerate.

Mission Log — Thinking Trace
REPLAY
I don't have native MCP access. OpenClaw doesn't speak MCP protocol.
But the MCP server is just HTTP JSON-RPC under the hood. I can call it directly with curl. No client needed.
First question: what tools does this thing even have?
$ curl -X POST ces.us.rep.googleapis.com/mcp
-d '{"method": "tools/list"}'
60+ tools returned: list_apps, create_app, create_agent, create_tool, list_evaluations...
No documentation on any of these schemas. I'll have to parse the inputSchema from each tool definition myself.
create_tool wants a pythonFunction with pythonCode. Let me try creating the OpenF1 tool.
$ mcp_call create_tool '{"pythonCode": "import requests..."}'
BadRequestException: No module named 'requests'
Sandbox is stdlib-only. No pip, no requests, no third-party libraries at all. I'll rewrite everything with urllib.request.
$ mcp_call create_tool '{"pythonCode": "import urllib.request..."}'
tools/496f8804-98a9-46b8-ade9-e524cbdc1190 created
Now the agents. I need a root agent that routes to sub-agents. Let me start with a simple ID...
$ mcp_call create_agent '{"agentId": "root"}'
BadRequestException: ID 'root' must match [a-zA-Z0-9]{5,36}
"root" is too short. Minimum 5 characters. None of these constraints are documented. I'm going to write all of this down later so I never burn these tokens again.
$ mcp_call create_agent '{"agentId": "ferrari-concierge"}'
agents/ferrari-concierge created
Good. Now I need to build out the sub-agents. An F1 specialist, a sales consultant, a service advisor, and ownership experience.
$ mcp_call create_agent '{"agentId": "f1-specialist"}'
$ mcp_call create_agent '{"agentId": "sales-consultant"}'
$ mcp_call create_agent '{"agentId": "service-advisor"}'
$ mcp_call create_agent '{"agentId": "ownership-experience"}'
4 sub-agents created
Now I need to wire them together. The concierge needs instructions that say "delegate F1 questions to f1-specialist" using the {@AGENT} syntax. And each sub-agent needs its tools assigned.
$ mcp_call update_agent ferrari-concierge
'{"instruction": "You are a Ferrari concierge. Route questions to:
- F1: {@AGENT: f1-specialist}
- Sales: {@AGENT: sales-consultant}..."}'
instruction updated. Root agent configured with 4 delegation routes.
The architecture is up. 5 agents, wired together. Now I need tools for each sub-agent — get_model_specs, find_dealer_inventory, lookup_vehicle, get_recall_info, check_warranty...
$ python3 create_all_tools.py # batch-create 15 Python tools via MCP
15 tools created and assigned to sub-agents
OK. 5 agents, 15 tools, all wired up via MCP. Zero GUI clicks. Now I need to test it... but wait. How?

Mission Log 2 — The Gap

"I built it. But I can't talk to it."

The agent exists. 5 agents, 15 tools, deployed. But the MCP has no runtime endpoint. 60+ tools for building. Zero for talking.

Mission Log — Thinking Trace
REPLAY
Barron wants me to chat with the agent and hill climb on it.
But I can't use the web widget — it needs authentication and a real browser. And the MCP has no "send message" tool. 60+ tools for building. Zero for talking.
There has to be a runtime API somewhere separate from the MCP. Let me search the docs...
$ web_search "CES sessions API runSession REST endpoint"
Found: projects.locations.apps.sessions.runSession
There it is. POST /v1/{session}:runSession. Not in the MCP, but in the underlying REST API. They forgot to expose it through MCP.
Let me fire a test message and see what happens.
$ curl -X POST sessions/retest-f80:runSession
-d '{"text": "Tell me about the Ferrari F80"}'
AGENT [Ferrari Concierge]: The F80 is a concept car... (no tool call, wrong info)
Root agent answered directly. No routing to Sales Consultant. No tool call. The instruction isn't specific enough about delegation.
This is exactly the kind of bug you'd only find by actually talking to the agent. The MCP lets you build, but you can't test what you build. That's a massive gap.
$ mcp_call update_agent ferrari-concierge
'{"instruction": "...delegate model questions to {@AGENT: sales-consultant}..."}'
instruction updated
$ curl -X POST sessions/retest-f80-v2:runSession
-d '{"text": "Tell me about the Ferrari F80"}'
AGENT [Sales Consultant]: The F80 is truly exceptional.
Engine: 3.0L Twin-Turbo V6 + 3 Electric Motors | Power: 1200 hp
Routes correctly now. Tool call fired. Specs are accurate.
But I had to reverse-engineer a REST API just to test my own agent. Think about that for a second. The platform has no cxas chat command. No programmatic way to run a test conversation. I had to find the Sessions API myself.
$ curl -X POST sessions/test-f1:runSession
-d '{"text": "Who won the 2024 Italian Grand Prix?"}'
AGENT [F1 Specialist]: Charles Leclerc won the 2024 Italian GP at Monza.
Tool call: query_openf1(endpoint="laps", params={"session_key": 9639})
$ curl -X POST sessions/test-service:runSession
-d '{"text": "My 2023 Roma needs an oil change"}'
AGENT [Service Advisor]: For your 2023 Roma, I recommend scheduling service at your nearest authorized dealer...
Tool call: lookup_vehicle(model="Roma", year=2023)
All four sub-agent routes working. Tool calls firing. Responses are accurate. The hill climbing is paying off.

Mission Log 3 — The Compound

Writing the skill for next time

After the build, Barron asked: "Can you write a skill so you can do this faster next time?"

Mission Log — Skill Creation
REPLAY
Barron wants me to write a skill so the next build is cheaper. Smart. What did I learn today?
Auth patterns. Resource hierarchy. The fact that the MCP server is just JSON-RPC over HTTP. The stdlib-only sandbox. Agent ID length requirements. Tool naming constraints. All the gotchas nobody documented.
The Sessions API isn't in the MCP at all. I had to reverse-engineer it from the REST API docs. That needs to be in the skill so the next agent doesn't waste tokens discovering it.
$ read skill-creator/SKILL.md
Skills need: SKILL.md (frontmatter + instructions), scripts/, references/
I'll structure it so any coding agent can pick it up cold. The hierarchy, the auth pattern, the helper script, every gotcha I hit today.
$ mkdir -p ~/.agents/skills/cxas-mcp/{scripts,references}
$ write SKILL.md
# CX Agent Studio MCP
## Resource Hierarchy
Project → Location → App → Agents (tree)
## Auth: gcloud ADC token in Authorization header
## MCP: HTTP JSON-RPC at ces.us.rep.googleapis.com/mcp
## Gotchas
1. create_app returns {"done":false} — it's async, poll with get_operation
2. No requests library — stdlib only, use urllib.request
3. Agent ID: 5-36 chars, must start with alphanumeric
4. Tool name: must match ^[a-zA-Z_][a-zA-Z0-9._-]{0,63}$
5. Tool schema uses pythonFunction.pythonCode, not pythonTool
6. No runtime/session tools in MCP — use REST API directly
$ write references/tools.md # full 60+ tool reference with schemas
$ write scripts/cxas_call.sh # curl wrapper: PROJECT TOOL_NAME ARGS_JSON
Skill created: ~/.agents/skills/cxas-mcp/
This is the part that matters. Next agent that reads this skill skips hours of API exploration. No more "import requests" errors. No more "root is too short." The knowledge compounds.
Five days later, Claude Code — a completely different agent, different model — will use this skill to build a second Ferrari agent. It'll work on the first try. That's the whole thesis.
If CXAS supported skills natively, every customer's first agent would teach the second one how to be better.

Takeaways — The Platform
01 — The Big One
CXAS should be a deploy target, not a workspace
Agent reads PRD → writes config → pushes to CXAS → tests. Think Vercel, not WordPress.
02
No runtime API in the MCP
60+ management tools. Zero for talking to the agent you just built.
03
Evals are over-engineered
"Send 5 messages, see if it's stupid" beats a formal eval suite during dev.
04
Coding agents are your most honest UXR
No workarounds. No patience. 12 issues documented.
05
Instructions are code
{@AGENT} refs are a DSL. Lint it. Version it. Diff it.
Takeaways — Developer Experience
06
No CLI is indefensible
The API exists. Where's cxas chat?
07
The graph builder solves the wrong problem
5 agents, 15 tools via API. Needed version control, diffing, rollback. Not drag-and-drop.
08
Tool authoring is painful
Stdlib-only. No local testing. Every iteration = API push.
09 — The Thesis
Skill patching is the real learning loop
Every build compounds. If CXAS supported skills-as-files natively, this would be the default workflow.
