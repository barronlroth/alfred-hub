CX Agent Studio · skill architecture

Two competing philosophies for agent-builder skills.

The question is not whether CXAS needs procedural knowledge. It does. The decision is whether that knowledge should appear to coding agents as many independent top-level skills, or as one compact router skill that loads focused reference files on demand.

Many small skills

school A

Each phase of the customer-service agent lifecycle becomes a separate discoverable skill. Useful when tasks are sharply bounded; noisy when every platform capability becomes another item in the global skill list.

One router skill

school B

One visible CXAS skill owns the platform surface. The top-level file acts like a router: detect intent, then load only the narrow markdown references needed for the task.

The tension

Small skills feel modular today, but every new capability competes for attention in a global agent skill index. That becomes painful when enterprise developers have dozens of tools beyond CXAS.

The router bet

A single CXAS skill can behave like a table of contents: cheap to discover, explicit about when to load detail, and hard to overflow.

What still splits out

  • Reusable cross-platform procedures
  • Heavy scripts or toolchains
  • Standalone skills useful outside CXAS
