Zero to Deployed

The developer journey from first line of code to production deployment is held together with zip files and prayer. This is the map out.

01
Layer 1

Code Sync

Get the agent into Git. The foundation everything else depends on.

Today

Zip files & prayer

  • Export from console → download zip → unzip → edit
  • Re-zip → import → creates a new app every time
  • No .gitignore, no project structure, no lineage tracking
  • Branching = export + import as new app + manually rename
Tomorrow

Pull, edit, push

# Clone agent to local files
cxas pull my-agent --project my-gcp-project

# Edit locally in your IDE
vim agents/concierge/instruction.txt

# Push changes back
cxas push ./my-agent/

# Branch with lineage tracking
cxas branch my-agent --name experiment-1

Deployment configs

Channel settings, widget customizations, deployment parameters — all console-only.

Callback hooks

Before/after agent, model, and tool callbacks don't round-trip through export/import.

Guardrail assignments

Active guardrails and their ordering relative to the app are absent from exports.

Console-only toggles

Advanced model params, logging settings, session TTL — invisible in exported files.

Resource lineage

No tracking of which GCP project, app ID, or branch a folder came from.

Environment bindings

environment.json is generated on export, must be manually edited per target.

Key insight

A developer can't git clone a repo and have a complete, deployable agent. There's always a gap that requires going back to the console.

02
Layer 2

Validation

Know it works before you push. Not after the console explodes.

Today

Import and pray

  • Edit a Python tool → import to console → discover forbidden import
  • Edit OpenAPI spec → import → discover invalid schema
  • Write callback hook → import → discover broken agent reference
  • Every error requires a full round-trip to surface
Tomorrow

Validate locally, fail fast

  • File structure: directories exist, naming conventions match
  • Python tools: valid syntax, stdlib-only imports, signature matching
  • OpenAPI toolsets: valid schema, well-formed endpoints
  • Cross-file references: tool/agent refs resolve to real files
  • Instruction XML: valid structure, valid {@AGENT} and {@TOOL} refs
  • Environment config: placeholders have corresponding values
Next step

VS Code extension with real-time linting. Red squiggles on forbidden imports, autocomplete for agent/tool references, syntax highlighting for instruction XML. The CLI validation logic, packaged for the IDE.

03
Layer 3

Hill Climbing

Iterate fast. Edit, eval, compare, improve. The inner loop that makes agents better.

Today

Fragmented iteration

  • Evals run manually in the console UI
  • Evals tied to a specific agent instance — they don't travel with code
  • Comparing versions = create second agent, replicate eval set, eyeball results
  • Can't branch an agent and its evals together
Tomorrow

Two paths to faster iteration

# Path A: Eval-driven hill climbing
cxas branch my-app --name experiment-1
cxas push ./experiment-1/
cxas eval run --agent experiment-1 --evals ./evals/
cxas eval run --agent my-app --evals ./evals/ \
  --compare experiment-1

# Path B: Conversational testing
# VS Code chat panel → sessions API
# Edit one pane, chat in another
Key insight

Evals should be their own set of code, decoupled from the agent. When evals live as files in Git, you can run the same eval set against multiple agent versions without replication, track eval changes in PRs, and never manually clone an eval configuration again.

04
Layer 4

CI/CD

Evals on every PR. Promotion on merge. The automation layer.

Today

Manual everything

  • Evals run by hand in the console — no connection to PRs
  • Promotion = export zip from dev, edit environment.json, import to prod
  • Config swap is manual — one mistake routes prod to dev endpoints
  • No automated gate between code review and eval results
Tomorrow

PR opens → evals run → results posted

  • GitHub Action pushes to temp branch agent via CLI
  • Eval suite runs automatically against the branch
  • Results appear as PR comment: pass/fail, metric deltas
  • Merge requires evals passing (branch protection)
  • CLI promote: automated export → env swap → import

Terraform's job

  • Create GCP projects
  • Enable CXAS APIs
  • Create app resources
  • Set up IAM bindings
  • Configure service accounts
  • Provision data stores

CLI's job

  • Pull/push agent code
  • Edit instructions and tools
  • Run evals
  • Promote agent content
  • Deploy versions
  • Manage traffic
Key insight

Terraform sets up the empty house. The CLI furnishes it. They're complementary, not competing. A GitHub Action is just "install CLI, run CLI commands, format output." The hard work is in the CLI; the CI integration is cheap.

05
Layer 5

Deployment & Traffic

Ship safely. Canary deploys. Traffic splitting. Insights-gated rollouts.

Today

All or nothing

  • Deploying a new version sends 100% of traffic immediately
  • No concept of percentage-based routing
  • No integration with live monitoring for auto-rollback
  • Versions and deployments exist — but no gradual rollout
Tomorrow

Progressive delivery

  • Phase 1: Traffic splitting between versions by percentage
  • Phase 2: Automated ramp schedule with Insights gates
  • Auto-rollback when session quality or error rates breach thresholds
  • Real user conversations as the signal source, not synthetic evals
v1 — 99%
v2
Current (stable) Canary (monitoring)
1% → 10% → 50% → 100% Gated on Insights: session quality, task completion, error rates
Why Insights, not evals?

Evals validate the agent before deployment. Once traffic is flowing, what matters is real-world performance — actual user conversations, actual task completions, actual error rates. Insights is the signal source for ramp decisions. Evals don't change during a rollout; live data does.

Open questions

01

Eval file format

Should the CLI adopt SCRAPI's format directly, or define a simpler YAML-based format that compiles down to it?

02

Branch cleanup

Temporary branch agents from CI should auto-delete. TTL? Explicit cleanup step? Both?

03

Multi-region

Currently US-only. When multi-region ships, every CLI command needs a --location flag. Design for this now.

04

Auth for CI

gcloud ADC tokens expire hourly. Service account key support needed for CI pipelines.
