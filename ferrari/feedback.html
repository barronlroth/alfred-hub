Issues encountered while building a multi-agent Ferrari customer support system entirely through the CXAS MCP server. Every agent, tool, toolset, and app was created programmatically — no console clicks. These are the rough edges.

#1 Python tool sandbox blocks all outbound HTTP Critical

What happened

Created a Python tool using urllib.request to call the OpenF1 API (api.openf1.org). The tool was created successfully but fails silently at runtime — no error message, just empty/timeout response.

Expected

Python tools should be able to make outbound HTTP requests, or the documentation should clearly state this restriction and recommend OpenAPI tools instead.

Impact

Forces developers to use OpenAPI toolsets for any external API integration, but that path has its own issues (see #2).

#2 OpenAPI Toolsets don't resolve through runSession API Critical

What happened

Created an OpenAPI toolset with 5 OpenF1 endpoints. Agent references it correctly. But calling runSession returns:

Function 'openf1:get_sessions' is not declared in the tools.

Expected

Toolset tools should be available to agents through all runtime paths, not just the simulator.

Impact

Completely blocks programmatic testing. Can't build CI/CD pipelines or automated testing for agents that use OpenAPI toolsets.

#3 create_tool rejects OpenAPI tools without helpful guidance High

What happened

Called create_tool with an openApiTool field. Got:

"Creating tools of type OpenApiTool is not supported. Please use OpenApi Toolsets instead."

Expected

Error should link to toolset documentation or provide a code example. Better: create_tool should accept OpenAPI tools directly.

Impact

Confusing DX. Developer has to discover toolsets exist through trial and error.

#4 No execute_tool capability in MCP High

What happened

After creating a Python tool, wanted to test it in isolation before wiring it to an agent. No execute_tool method exists in the MCP server.

Expected

MCP should expose tool execution for testing/debugging without requiring a full agent conversation.

Impact

Slows development loop. Every tool test requires creating an agent, starting a session, and crafting a prompt that triggers the tool.

#5 Tool displayName is silently read-only Medium

What happened

Set displayName when creating a Python tool. No error returned, but the value was ignored — displayName is auto-derived from the tool type.

Expected

Either accept and use the provided displayName, or return a validation error explaining it's read-only.

Impact

Minor confusion, but violates principle of least surprise.

#6 Agent ID minimum length not documented Medium

What happened

Tried to create an agent with ID "root". Got validation error about minimum length (5 chars). The constraint isn't in the MCP tool schema description:

[a-zA-Z0-9][a-zA-Z0-9-_]{4,35}

Expected

Schema description should include ID format constraints, or accept shorter IDs.

Impact

Trial and error to discover valid ID format.

#7 create_app returns async operation with no tracking Medium

What happened

create_app returns {"done": false} with no operation ID or completion callback. Had to poll list_apps to confirm creation.

Expected

Return an operation ID that can be polled, or complete synchronously for simple creates.

Impact

Race conditions possible in automated pipelines.

#8 No conversation capability in MCP server High

What happened

Built an entire agent via MCP, then realized there's no way to chat with it through MCP. Had to discover the separate ces.googleapis.com REST API with runSession.

Expected

MCP server should include a run_session or send_message tool for testing agents inline with the build workflow.

Impact

Breaks the build-test loop. Coding agents (Codex, Claude Code) can build agents via MCP but can't test them without switching to a completely different API.

#9 Web widget JS/CSS versioning mismatch High

What happened

Documentation references v1/ paths for chat-messenger JS and CSS. The console's embed code generator produces v1.12/ paths. Using the docs version = widget doesn't render, no error in console.

Expected

Docs should always reference the latest versioned path, or v1/ should be an alias for latest.

Impact

Silent failure. Developer deploys widget, nothing appears, no way to diagnose without comparing against console output.

#10 chat-messenger-layout.css breaks host page scroll High

What happened

Included chat-messenger-layout.css as recommended in the embed code. It sets overflow: hidden on the body element, completely breaking scroll on mobile.

Expected

Widget CSS should be scoped to the widget element, not modify global body styles. At minimum, the docs should warn about this more prominently.

Impact

Breaks the host website. Required debugging to identify the cause.

#11 No requests module in Python sandbox Medium

What happened

Wrote a Python tool using import requests (the most common Python HTTP library). Failed at runtime with import error.

Expected

Either include requests in the sandbox, or prominently document the available modules in the Python tool creation flow (not buried in a reference page).

Impact

Every Python developer's first instinct is import requests. Failing here wastes time.

#12 Toolset tool names are prefixed at runtime Medium

What happened

Created a toolset with operationId get_sessions. At runtime, the agent tries to call openf1:get_sessions (prefixed with toolset name). Instructions referencing the operationId directly don't work.

Expected

Document the naming convention for toolset tools, or allow unprefixed references.

Impact

Instructions that reference tools by operationId fail. No documentation explains the prefixing behavior.
