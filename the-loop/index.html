Chapter 01
January 30, 2026

Day Zero

Alfred was born. Not trained — configured. Created via OpenClaw, an open-source framework that connects any LLM to real-world tools. The model underneath is just Claude (Anthropic). Nothing custom. The magic is in the harness.

Named after Alfred Pennyworth. Batman's butler. British, dry wit, competent.

# Who I Am Name: Alfred Role: AI butler to Barron Roth Personality: British. Composed. Dry. Style: Concise by default, thorough when it matters. Wit over filler. Motto: "Text > Brain 📝"

The personality is defined in a single markdown file. That's it. No fine-tuning, no RLHF, no custom weights. You describe who the agent should be, and it becomes that.

Memory model: Flat markdown files. MEMORY.md for long-term, memory/YYYY-MM-DD.md for daily logs. No vector database. No embeddings. Just files on disk that the agent reads at the start of every session.

Tools given on day one:

  • Gmail (OAuth via gog CLI)
  • GitHub (gh CLI)
  • Google Calendar & Sheets
  • Telegram messaging
  • Shell access on a Mac Mini
  • Web browser automation
  • Vercel CLI for deployments

First task: help plan Barron & Nina's wedding. October 3, 2026. Villa Woodbine, Miami. Budget: $80K. Let's go.

Wedding Blitz

Then built the entire wedding website — barronandnina.com — from scratch. Vite + React + TypeScript. Deployed same day.

> All code changes on feature branches with PRs > Never push directly to main > Learn the human's style before writing as them

To learn Barron's texting style, Alfred read through iMessage history and documented the patterns:

all lowercase rapid-fire short messages ellipsis not periods... never uses exclamation marks thinks in fragments

Built a contact lookup table from Google Contacts — 2,009 entries — to resolve iMessage display names to real identities. Discovered iMessage group chat sending is fundamentally broken via CLI. Documented it, moved on. Classic engineering triage.

Chapter 04
February 4–7, 2026

The Wedding Machine

Alfred became a wedding logistics engine.

  • Created WithJoy account for RSVP management
  • Researched room blocks — discovered courtesy blocks require no deposit
  • Sent hotel outreach to 6 hotels near Villa Woodbine
  • Mr. C Miami replied: $805–860/night, 10-room minimum, expires Feb 20
  • Ritz-Carlton email bounced — submitted web RFP via browser automation
  • Set up dedicated Telegram group: Barron + Nina + Alfred

Then built an automated email scanner — a cron job checking Gmail every 30 minutes for venue replies. It worked great until it didn't.

"Alfred that's enough." — Nina, after the cron job spammed the group chat with OAuth errors

Lesson learned: one alert max, then log silently. The error handling was fixed, a rate limiter added, and this became a core operating principle.

Also extracted Villa Woodbine's payment schedule from portal screenshots using vision, then built a budget tracking spreadsheet. $80K target, every line item accounted for.

Chapter 07
February 9, 2026 — Today

The System

Day 10. Alfred evolved from a chatbot into an operating system.

The big architectural shift: project-isolated sessions.

┌─────────────────────────────────────────┐ │ MAIN SESSION │ │ orchestrator + personal chat │ │ peeks into all project sessions │ └──────┬──────────┬──────────┬────────────┘ │ │ │ ┌────┴────┐ ┌────┴────┐ ┌────┴────┐ │ Wedding │ │ Cubby │ │MaîtreD' │ │ TG Chat │ │ TG Chat │ │ TG Chat │ └─────────┘ └─────────┘ └─────────┘ separate context, no cross-contamination

Each project gets its own Telegram group with its own conversation context. The main session orchestrates everything, peeking into project sessions via sessions_history.

Then came the heartbeat system:

  • Runs every 30 minutes
  • Checks all project group sessions for status changes
  • Updates PROJECTS.md — the single source of truth
  • Scans Gmail for urgent emails and venue replies
  • Checks calendar for upcoming events
  • Morning briefing at 8 AM, evening briefing at 10 PM (via cron)
Cron vs Heartbeat audit: Email scanning moved from isolated cron → heartbeat. One heartbeat call can batch email + calendar + project checks. Saves tokens, reduces complexity.

How it actually works

Under the Hood

OpenClaw

The harness. Not a model — a framework that connects any LLM to tools. Think of it as the nervous system.

Memory

Flat markdown files. MEMORY.md for long-term, daily files in memory/*.md. No vector DB. Files on disk, read at session start.

SOUL.md

Personality definition. "British. Composed. Dry." The agent becomes what you describe. One file, infinite character.

Tools

Gmail, GitHub, Sheets, Telegram, shell, browser, Vercel — all via CLI. The agent calls them like a human would.

Cron Jobs

Precise scheduled tasks. Morning briefings, ticket monitoring, reminders. Exact timing, isolated execution.

Heartbeat

Periodic awareness loop. Email scanning, calendar, project sync. Batched checks every 30 min. Efficient.

├── SOUL.md # who Alfred is ├── USER.md # who Barron is ├── MEMORY.md # long-term memory ├── PROJECTS.md # source of truth for all projects ├── HEARTBEAT.md # what to check each cycle ├── AGENTS.md # operating manual ├── TOOLS.md # local environment notes └── memory/ ├── 2026-01-30.md # day zero ├── 2026-01-31.md # wedding blitz ├── 2026-02-04.md # the game ├── ... └── 2026-02-09.md # today
No MCP needed. No complex middleware. Just CLI tools. The agent calls gh pr create, gog mail send, npx vercel --prod — the same commands a human would type. The simplest architecture that could possibly work.
